# safe-commit အသုံးပြုနည်း

Feature တစ်ခု ရေးဆွဲပြီးစီးချိန်မှာ ကုဒ်တွေကို မလိုအပ်တဲ့ လျှို့ဝှက်ချက်တွေ (Sensitive data) မပါသွားအောင်၊ Format သပ်ရပ်အောင်၊ Test တွေအောင်မြင်အောင်၊ မှတ်တမ်းနဲ့ Document တွေ Update ဖြစ်အောင် သေချာစစ်ဆေးပြီးမှ Commit နဲ့ Push လုပ်ပေးမယ့် အဆင့်ဆင့်ပါဝင်တဲ့ Master Workflow Skill ဖြစ်ပါတယ်။ အဆင့်တိုင်းမှာ လူသား (User) ရဲ့ အတည်ပြုချက် (Approval) ယူပြီးမှသာ ရှေ့ဆက်ပါမယ်။

## အလုပ်လုပ်ပုံအဆင့်ဆင့် (Steps)

**အဆင့် ၁: Data Safety Check (လုံခြုံရေး စစ်ဆေးခြင်း)**
- လျှို့ဝှက်ကုဒ်တွေ (`.env`, `API keys`, `.db`, credentials) ပါသွားနိုင်ချေ ရှိမရှိ စစ်ဆေးပါမယ်။ တွေ့ရှိပါက User ကို အသိပေးပြီး မသိမ်းဆည်းအောင် (untrack လုပ်ရန်) အတည်ပြုချက် တောင်းခံပါမည်။

**အဆင့် ၂: Auto-Formatting (ကုဒ်များ သပ်ရပ်အောင်ပြင်ဆင်ခြင်း)**
- `src/` အောက်က ပြောင်းလဲထားတဲ့ ဖိုင်တွေကို Prettier (`npx prettier --write`) ဖြင့် အလိုအလျောက် သပ်ရပ်အောင် ပြင်ဆင်ပါမည်။

**အဆင့် ၃: Pre-check (Lint)**
- `src/` အောက်က ပြောင်းလဲထားရင် `npm run lint` ကို Run ပါမယ် (ဒီ project မှာ test runner မထည့်ရသေးလို့ lint ပဲ ရှိပါတယ်)။ မှားယွင်းမှုတွေရှိရင် ရပ်တန့်ပြီး ပြင်မလား၊ ကျော်သွားမလား (Bypass) မေးပါမယ်။

**အဆင့် ၄: Auto `/update-docs` (Document များ အပ်ဒိတ်လုပ်ခြင်း)**
- Architecture Document တွေ ပြင်ဆင်ဖို့ လိုမလို စစ်ဆေးပြီး လိုအပ်ပါက အတည်ပြုချက်တောင်း၍ ပြင်ဆင်ပါမည် (သို့မဟုတ် Skip လုပ်နိုင်ပါသည်)။

**အဆင့် ၅: Auto `/save-memory` (မှတ်တမ်းများ သိမ်းဆည်းခြင်း)**
- ရှေ့ဆက်မယ့် အလုပ်တွေအတွက် ဆုံးဖြတ်ချက်နဲ့ အကြောင်းအရာတွေကို `MEMORY.md` နှင့် ဆက်စပ်ဖိုင်တွေမှာ မှတ်တမ်းတင်ဖို့ အတည်ပြုချက်တောင်းခံပါမည်။

**အဆင့် ၆: Code Review & Atomic Commits (ကုဒ်စစ်ဆေးခြင်းနှင့် သေချာစွာ မှတ်တမ်းတင်ခြင်း)**
- ပြင်ဆင်ထားသမျှကို Deep Review စစ်ဆေးပြီး အပိုင်းလိုက် (Logical Grouping) ခွဲခြားပါမယ်။ Group တစ်ခုချင်းစီအတွက် သင့်လျော်တဲ့ Commit Message ကို အင်္ဂလိပ်/မြန်မာ နှစ်ဘာသာဖြင့် တင်ပြပြီး အတည်ပြုချက်ရမှသာ `git commit` အသီးသီး ပြုလုပ်ပါမည်။

**အဆင့် ၇: Post-Commit Sync (အွန်လိုင်းပေါ်တင်ခြင်း)**
- Commit အားလုံး အောင်မြင်ပါက အွန်လိုင်း (Remote repo) ပေါ်သို့ `git push` ဖြင့် တင်ရန် အတည်ပြုချက် တောင်းခံပါမည်။
